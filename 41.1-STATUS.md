# Shen 41.1 Port Status for shen-lua

This document describes the current state of the Shen 41.1 port (targeting `ShenOSKernel-41.1/klambda`).

## What Works

- The full 41.1 KLambda kernel loads successfully:
  - All 21 `.kl` files (core + `stlib`, `compiler`, `extension-*`, etc.).
  - `shen.initialise` completes without fatal error.
  - `version` reports `"41.1"`.
- Basic primitives function (`+`, `cons`, `load`, `eval-kl`, etc.).
- Simple KLambda evaluation works via `run_kl_string` / `eval`.
- Platform globals are set (`*language*`, `*implementation*`, `*port*`, `*porters*`, `*os*`, `*release*`).
- Compiler/runtime mitigations for 41.1 realities:
  - Avoided "more than 200 local variables" in giant `do` chains (`stlib.initialise-arities`, `initialise-lambda-forms`, etc.).
  - Generalized large literal data hoisting (`KDATA`) for embedded source trees inside `shen.record-kl` calls and similar.
- KL directory discovery is flexible (prefers 41.1 layout; also finds legacy `../shen-c` clones for comparison; respects `SHEN_KL_DIR`).
- A convenience test driver (`run-41.1-tests.lua`) exists for reproducing the official suite.

The port can boot a live 41.1 Shen environment.

## What Does Not Work Yet (Major Gaps)

### 1. Macro / `*macros*` Initialization (Critical Blocker)
After `shen.initialise`, the global `*macros*` is **not populated** in `GLOBALS` (the value the rest of the system and user code expects).

- Some globals appear under `shen.*` prefixes (`shen.*tc*` etc.) while others are bare.
- `*macros*` (and bare `*tc*`) end up missing or `nil`.
- This is required very early by the macro expander, `defmacro`, package forms, and the test harness.

We had to manually seed `*macros*` in the test driver just to get past the first error ("variable *macros* has no value").

The root cause appears to be in how `shen.initialise-environment` (the giant nested `do` in `init.kl`) executes under our compilation of `shen.initialise-environment`. Later globals (`*property-vector*`, `*version*`) get set; the `*macros*` set (and some earlier ones) do not take effect in the observed `GLOBALS` table.

### 2. Test Harness Fails Immediately
Even with the `*macros*` seed:

- `(load "harness.shen")` starts (package + some definitions) but then fails with:
  ```
  attempt to apply a non-function
  ```
- This happens inside the macro machinery / `defmacro` expansion / `exec-macro` / lambda-form lookup used by the harness (`report`, `rcons`, etc.).
- Consequently, `kerneltests.shen` (and thus the full official suite via `runme.shen`) never executes any `(report ...)` blocks.
- No pass/fail counts or "pass rate" output is produced.

The harness is the standard way to run the kernel test suite (`tests/` directory). Until this works, we cannot claim certification or even meaningful coverage for 41.1.

### 3. `fn` / Lambda-Form / Higher-Order Function Issues
41.1 makes heavier use of:
- `(fn F)` (and the `function` wrapper)
- `shen.set-lambda-form-entry` / lookup in `*property-vector*` under `shen.lambda-form`
- Closures registered during `shen.initialise-lambda-forms`

Our compiler has special handling for `lambda`/`freeze`, and the runtime supports currying/`APP`/`thaw`, but the interaction with the kernel's lambda-form table + `fn` (especially for 0-arity and macro/prolog cases) is not yet fully compatible. The "non-function" error above is a symptom of this.

### 4. Other Observed / Likely Issues
- Some early kernel code and stlib assume globals and side effects from the full `shen.initialise-*` + `stlib.initialise` sequence that our load order + compilation may not perfectly replicate.
- Typed mode (`tc +`) sections in the tests (N-Queens, search, interpreter, proof assistant, etc.) have not been exercised.
- Prolog, YACC, and the larger `stlib` components (maths, strings, vectors, IO, etc.) are loaded but untested at scale.
- Interactive bits in the harness (`y-or-n?` on failure) and any REPL/launcher behavior from the new extensions are not validated.
- Performance / startup numbers from the old 22.4 benchmarks are obsolete; `stlib` adds significant load time and compiled code size.

## Current Workarounds in the Codebase
- Compiler changes are conservative and only affect codegen for large `do` forms and constant data (safe for 22.4 too).
- `boot.lua` now seeds platform globals and has fallback discovery for both 41.1 and legacy layouts.
- The test driver (`run-41.1-tests.lua`) contains the seeding hack + step-by-step loading + error extraction for diagnosis.

These are **not** intended as permanent fixes; they paper over initialization and higher-order call gaps.

## How to Reproduce the Current State
```sh
# From shen-lua dir, with 41.1 tree at ../cl-source/ShenOSKernel-41.1
luajit -e '
  local P = require("boot")
  P.load_kernel(false)
  P.initialise()
  print("version:", P.F["version"]())
'
# To attempt the official suite (will fail early):
luajit run-41.1-tests.lua
```

Use `SHEN_KL_DIR=../shen-c/shen/src/kl ...` if you have the legacy shen-c clone for 22.4 comparison.

## Next Steps (Rough Priority)
- Investigate why the `(set *macros* ...)` (and nearby sets) inside the compiled `shen.initialise-environment` body do not result in entries in `P.GLOBALS`.
- Fix / extend lambda-form registration + the `fn` implementation so that macro functions and 0-arity entries are properly callable.
- Make global lookup (`value` / bare vs. `shen.` prefixed names) more robust during early initialisation.
- Get the test harness to load and run at least the first few `(report ...)` blocks, then the full `kerneltests.shen`.
- Re-run the official suite and collect real pass/fail rates.
- Consider caching compiled kernel images (the 41.1 stlib makes cold start noticeably heavier).
- Update / re-generate `BENCHMARKS.md` once a working baseline exists.
- Handle any new primitives or reader changes that 41.1 introduced (currently none obvious, but the `fn` path and stlib IO helpers are exercised more).
